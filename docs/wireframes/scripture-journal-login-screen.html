Annotations — Login Screen

  1. [1] No navigation header — The Login Screen has no global nav bar. It is the only unauthenticated screen in Phase 1 (FR-97). The app logo appears inside the login card as the sole brand anchor. Adding a nav bar would imply the user has access to routes they do not.
  2. [2] Error message is deliberately vague — "Invalid username or password." Never specifies which field is wrong. This prevents username enumeration — an attacker cannot distinguish "that username doesn't exist" from "that password is wrong." Security requirement from Flow 7. Both fields receive the error border to reinforce that either could be the problem. NFR-17.
  3. [3] Username is retained on error; password is cleared — After a 401, the username field keeps its value so the user doesn't have to retype it. The password field is cleared, matching standard browser and native app conventions (prevents the plain-text password lingering in an input). The password field placeholder "••••••••" appears again to signal it is empty.
  4. [4] Rate limit (429) disables the form — When the server returns 429, both fields and the submit button are disabled. The amber warning message instructs the user to wait. This is a distinct visual state from the red 401 error — it communicates "the system is throttling you" rather than "your credentials are wrong." Flow 7, NFR-17.
  5. [5] Session-expired state uses a blue informational alert — When the user is redirected here because their JWT expired mid-session, the `?return=` query param is present in the URL and a blue info notice explains what happened. The form fields are empty (username not pre-filled — the user may want to log in as a different account). After successful login, the app redirects to the `return` destination. Flow 7, FR-04.
  6. [6] No "Forgot password" link — FR-95 explicitly excludes password reset via email (no email infrastructure in Phase 1). The footer note directs users to the administrator for account issues. Admin manually resets passwords via the admin CLI endpoint. This is intentional and should not be added as a "quick win" — it would require email infrastructure.
  7. [7] No "Sign up" or "Register" link — FR-06 specifies that accounts are admin-created only via CLI or admin endpoint, not a public form. The footer note ("Contact the administrator") is the correct affordance. A registration link would be misleading and non-functional.
  8. [8] Loading state disables all inputs and button — During the sign-in request, both fields and the button are disabled to prevent double-submit. The button label changes to "Signing in" with an animated dots indicator. FR-60.
  9. [9] Centered card layout — no surrounding chrome — The login card is centered horizontally and vertically on a gray background. There is no sidebar, header, or footer outside the card. This is the simplest possible layout — login is a single-purpose screen. Desktop only (FR-96), so a fixed 380 px card width is appropriate.
  10. [10] Password field uses browser-native masking — The wireframe shows "••••••••" as a placeholder convention. In production this is an <input type="password"> — the browser handles masking, show/hide toggle (browser-provided in modern browsers), and autofill. No custom show/hide toggle is designed here, but the browser's native one is acceptable.
  11. [11] Return URL preserved in the query string — When the router detects an unauthenticated visit to a protected route, it redirects to /login?return=<encoded-path>. After successful login, React Router reads this param and navigates to the original destination. The URL is visible in State 5's browser chrome to illustrate this. Flow 1, Flow 7.
UX review checklist
  • ✅ Primary action (Sign In button) is the only action — visually dominant, full width
  • ✅ Error message does not reveal which field is wrong — username enumeration prevented
  • ✅ Rate limit is a distinct state from invalid credentials — different color, disables form
  • ✅ Session-expiry redirect explained clearly with return-to destination preserved
  • ✅ Loading state prevents double-submit
  • ✅ No "Forgot password" or "Sign up" — both are correctly absent per FR-95, FR-06
  • ✅ No navigation header — unauthenticated screen has no routes to offer
  • ⚠️ No "Cancel" or back button — there is nowhere to go back to on an unauthenticated visit. This is correct; note it so it is not added by mistake later.
