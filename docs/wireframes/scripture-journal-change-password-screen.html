Annotations — Change Password Screen

  1. [1] Entry point: user menu dropdown, not the nav bar — The screen is reached by clicking "Change Password" in the username dropdown in the top-right of the nav. This is the only route in. The dropdown also contains "Log Out." The entry-point state shows the dropdown open so this path is explicit in the wireframe. Route: /settings/password. Flow 6.
  2. [2] Full nav is present — authenticated screen — Unlike Login, this screen has the full global nav because the user is authenticated. "Browse Scripture" and "Import Article" remain accessible. The user can abandon the password change at any time without consequence. FR-07.
  3. [3] Three fields, two logical groups — A visual divider separates "Current password" (identity verification) from "New password + Confirm" (the new credential). This grouping makes the two-step intent explicit: first prove who you are, then set the new value. The group labels use a small uppercase caption to avoid adding a third card section.
  4. [4] Client-side validation before any server round-trip — Two errors can be caught before submitting: (a) "Passwords do not match" — new password ≠ confirm field; (b) "New password must be different from your current password" — new password field == current password field (comparing plaintext inputs; the server also verifies this). Errors appear inline beneath each field on submit attempt, not on blur. Flow 6, NFR-13.
  5. [5] Server-side error: only the current password field is cleared — When the server returns a 401 (wrong current password), the current password field is cleared and receives the error state. The new password and confirm fields retain their values — the user typed a good new password; there's no reason to make them retype it. The inline hint below the new password field says "New password is preserved — only re-enter your current password above." FR-61.
  6. [6] Success: stay on the same screen, reset the form — On success, a green alert appears at the top of the card explaining the password is active and the current session remains valid. The form is reset to empty so the user could immediately change it again (unlikely but possible). No forced redirect or logout. Flow 6: "JWT remains valid — no forced re-login."
  7. [7] JWT is NOT invalidated on password change (Phase 1) — The success message explicitly states "Your current session remains valid." This is a deliberate Phase 1 decision from Flow 6. It means a session that was hijacked before the password change continues working — acceptable for a single-user personal tool. This should be noted in the spec as a known trade-off for future tightening.
  8. [8] Loading state disables all fields and both buttons — All three password fields, Cancel, and the submit button are disabled during the save. The button label changes to "Saving" with animated dots. The back link is also dimmed (non-interactive) to prevent navigating away mid-request. FR-60.
  9. [9] Cancel and "← Dashboard" both discard and return — The "← Dashboard" back link and the "Cancel" button in the footer both navigate back to / without saving. The form has no auto-save or draft state. Two affordances for the same action because "back link" (top-left of footer) and "Cancel button" (bottom-right) are familiar conventions in different visual positions — both should work. No confirmation dialog is needed since no data has been saved yet.
  10. [10] No password strength meter in Phase 1 — A minimum length hint ("at least 8 characters") is shown below the new password field, but there is no real-time strength meter or character requirement checklist. This keeps the form simple. The 8-character minimum is enforced server-side via Zod validation (NFR-13) and reported as an inline field error if violated.
  11. [11] Admin password reset is out of scope for this screen — FR-95 excludes email-based password reset. If a user forgets their password and cannot log in, the admin resets it via CLI — that is not a UI flow. This screen is only for authenticated users who know their current password and want to change it.
UX review checklist
  • ✅ Primary action (Update Password) is visually prominent — dark full-width-of-footer button
  • ✅ Two logical groups (verify identity / set new password) clearly separated
  • ✅ Client-side validation catches mismatch and same-as-current before server round-trip
  • ✅ Server error (wrong current password) only clears the current password field — new password preserved
  • ✅ Success state stays on page, explains JWT remains valid, resets form
  • ✅ Loading state prevents double-submit and back-navigation mid-request
  • ✅ Cancel / ← Dashboard available in all non-loading states
  • ✅ No destructive action on this screen — password change is reversible by changing again
  • ✅ Entry point (user dropdown) shown explicitly so the navigation path is documented
