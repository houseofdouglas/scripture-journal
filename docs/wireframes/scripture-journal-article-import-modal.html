Annotations — Article Import Modal

  1. [1] Modal as overlay, not a page — The modal appears as an overlay on top of whichever screen the user came from (typically the Dashboard). The background is dimmed but visible. Pressing the ✕ or "Cancel" dismisses the modal and returns focus to the underlying page. There is no dedicated route for the modal itself — it's triggered by clicking "Import Article" in the nav. FR-20.
  2. [2] Allowlist hint is always visible — The input hint ("Only articles from churchofjesuschrist.org are supported") appears below the URL field in all states, not only on error. This sets expectation before the user tries an unsupported URL. The error state (State 3) repeats the constraint more prominently with a red inline message. FR-20.
  3. [3] Cancel is always available (except during fetch) — The ✕ and "Cancel" button are disabled during the fetch (State 2) to prevent partial state. In all other states they are active and dismiss the modal cleanly. FR-60.
  4. [4] Loading state disables all controls — While fetching, the URL input, buttons, and ✕ are all disabled / dimmed. The progress bar is indeterminate (CSS striped pattern) since we can't know exactly how long the strip-and-hash step will take. The modal is not dismissible by clicking the backdrop during this state. FR-60.
  5. [5] Fetch failure clearly offers two exits — State 4 gives the user two clear choices: (a) edit the URL and try again (primary button reads "Try Again", not "Import"), or (b) follow the underlined "Paste article text manually instead →" link to switch to State 5. The attempted URL is preserved in the input so the user doesn't have to retype it. FR-24.
  6. [6] Manual paste preserves the source URL — State 5 shows a read-only display of the attempted URL above the title and textarea fields. The URL will be stored in the article's metadata even though it was never successfully fetched — it records the user's intent and the origin. FR-25.
  7. [7] Manual paste uses double-newline paragraph splitting — The placeholder text in the textarea explains this convention ("Separate paragraphs with a blank line"). There is no rich-text input — plain text only. Each paragraph becomes a separately annotatable block (blockId = zero-indexed paragraph number). FR-21, Flow 8.
  8. [8] "← Back to URL" in manual paste — The left-side footer button in State 5 returns the user to the URL input (State 1) with their URL preserved. This is an escape hatch if they launched manual paste accidentally. It replaces the "Cancel" button's position; a separate "Cancel" (dismiss modal entirely) is not shown here to avoid three-button footer clutter — ✕ in the header still closes the modal.
  9. [9] Duplicate warning shows article metadata — State 6 displays the article's title and a compact metadata row (source, date, paragraph count). This lets the user confirm they are looking at the right article before choosing. The warning tone is amber (non-destructive) since both options lead safely to the Article View. FR-23.
  10. [10] "Re-import" on a duplicate is a no-op — communicated clearly — The warning notice explains that identical content means no new version is created. "Re-import" simply navigates to the existing Article View. This prevents user confusion ("did it import or not?"). FR-22.
  11. [11] New version notice is informational blue, not amber warning — State 7 uses a blue notice because creating a new version is the expected happy-path behaviour for changed content — it's not a warning. The prior version's date is shown so the user understands the timeline. The explicit "Create New Version" button requires user confirmation before any write occurs. FR-27.
  12. [12] Both choices on the duplicate screen lead to the same Article View — "Open Existing Copy" and "Re-import" both navigate to /articles/:articleId. The articleId is the same SHA-256 in both cases, so the destination is identical. The only difference is user intent / reassurance. FR-22, FR-23, Flow 9.
  13. [13] Modal title changes per state — "Import Article" (default/loading/error) → "Paste Article Text" (manual paste) → "Article Already Imported" (duplicate) → "Article Has Changed" (new version). The title tracks context so the user always knows what decision they are making.
UX review checklist
  • ✅ Primary action (Import button) is visually prominent in every state
  • ✅ Allowlist constraint communicated before and after error — not only on failure
  • ✅ Loading state prevents accidental dismissal and shows progress
  • ✅ Fetch failure offers fallback path (manual paste) without dead-ending the user
  • ✅ Duplicate and new-version states give enough context to make an informed choice
  • ✅ Cancel / ✕ is always available (except during active fetch)
  • ✅ No destructive action on this modal — all paths either import or navigate to existing content
  • ✅ Source URL preserved through manual paste flow (FR-25)
  • ✅ Article metadata (title, date, paragraph count) shown in duplicate warning so user can confirm identity
